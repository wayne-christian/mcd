Regardless of whether you are using compute through PaaS (Platform) or IaaS (Infrastructure), your cloud service will be running on a virtual machine

For PaaS, the virtual machine is abstracted away and you just manage the data and application code, hence it is called a managed service

For IaaS, you manage the virtual machine, and with that have full access to the VM’s OS and responsibility for updates etc

The hint here is the terminology: instance, a single virtual machine is known as an instance

An instance can be configured, e.g. number of cpus, RAM, disk size etc

You should have already configured compute instances for both PaaS and IaaS using yaml configuration files

Virtualisation

Virtualisation

Virtualisation has been around for some time

In the 70s IBM pioneered hardware-level virtualisation (http://www.vm.ibm.com/vm40hist.pdf)

The key benefit back then, as it still is now, was to run multiple operating systems on the one physical machine

Virtualisation

Nowadays virtualisation of servers helps to prevent under-utilisation of traditional on-premises compute and datacentre resources

It allows rapid testing and development of specific server images

Supports business continuity by encapsulating entire systems into single image files – e.g. Virtual Hard Disk (VHD)

These can be restored anywhere (i.e. to cloud)

What is a Virtual Machine?

“A virtual machine (VM) is an operating system OS or application environment that is installed on software which imitates dedicated hardware. The end user has the same (almost!) experience on a virtual machine as they would have on dedicated hardware.”

You have probably used one before…

Virtualisation – Hardware Acceleration

Hardware acceleration features for VMs are now ubiquitous

Provide performance enhancements by allowing direct access to hardware

Virtualisation – Hardware Acceleration

Intel’s Virtualisation (VT-x) is built into its newer x86 CPUs:

Provides specific x86 CPU hardware instructions to manage VMs (faster)

Without this, all VM management is done purely through software (slower)

Intel’s Graphics Virtualisation Technology (GV-T)

Recently introduced as Intel Iris Pro

Allows the on-die GPU (Circuits on the same chip) to be dedicated to one or more VMs

Physical machine can also have a *discrete GPU

*A discrete GPU is a Graphics Processing Unit that is not part of the CPU, ie, it's a separate add in board or a separate chip on the system board (usually in laptops and AIOs.) Discrete GPUs also use their own VRAM instead of using system RAM

Virtualised phone emulators for Windows, Android and iOS are good examples of CPU hardware acceleration

Without it they would be painfully slow (Android??)

Allows developers to get realistic ‘like-for-like’ performance that nearly mirrors physical device performance

Virtualisation Approaches

There are two main approaches to virtualisation in x86 systems:

Hosted architecture

Hypervisor (bare-metal) architecture

Hosted

Hosted has better compatibility across different types of hardware, it is installed on top of the host OS

Hosted is generally directed at the individual end-user and usually comprised of low cost or ‘free’ software i.e. Virtual Box, VMware player

Hyper V

Hypervisor has greater scalability, reliability, and performance, it is installed directly over the hardware and part hardware device drivers

Hypervisor is the approach used by cloud vendors for their Cloud OS

Hypervisor (bare-metal) architecture

Hosted Architecture

Hosted virtualisation runs ‘on top’ of the host OS as a guest OS

Guest OS is abstracted away using VM software such as Virtual Box or VMware workstation

Type of approach you will use to install personal or small group installations of Windows on Linux and vice-versa

Hosted Architecture (limitations)

Does not normally have direct access to physical hardware

Access to hardware features varies greatly between VM software vendors, and may depend on CPU hardware virtualisation features.
For example virtual serial or USB ports (from host OS) may not be available

Virtualisation – Hosted Single User

In cloud virtualisation this layer (red box) is commonly referred to as the hypervisor, different vendors use different types, more on this later.

Bare-metal Virtualisation

Hypervisor (Bare-metal) Architecture

Multiple hardware (a Server Blade, a server farm or a data center)

Share hardware by dividing resources (CPUs, RAM, hard disks, network)

A Virtual Machine is an isolated software container which runs its own operating system, independent of the underlying hardware, and applications behaving like they are running on a physical computer.

In the cloud, VMs reside on top of hypervisors which give direct access to the hardware for optimal performance

Hypervisor (Bare-metal) Architecture

The virtualisation software (Hypervisor) is installed on the bare-metal physical machine first (Server Blade),
before any guest OS

Sits on top of hardware device drivers

Direct access to hardware resources

Each guest OS runs as a process (a VM instance)

Scales with great performance

Cloud Operating Systems

Cloud Operating System - Azure

It is important to note that a cloud platform is not just running on a cluster of virtualised Microsoft Server/Linux OS machines

Cloud providers (i.e. Amazon, Azure, Google) will have their own Cloud OS ‘agents’ that manage their infrastructure across many services (controllers)

Fabric Controller (FC)

For Azure/Google it is called the Fabric Controller (FC) - where fabric is an abstraction of resources (PaaS services, VMs, etc.) that are dynamically created on demand via a Hypervisor service

The Hypervisor is an agent/service of the FC

All cloud compute, data, and networking resources are managed by the FC

For example when you deploy a VM on Azure the FC takes care of this for you and does all required configuration and deployment to the server rack.

IaaS Virtualisation

Generally, Cloud IaaS virtualisation is of the native hypervisor (bare-metal) type in cloud server virtualisation, for scalability and performance
